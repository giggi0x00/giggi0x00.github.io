Bypass STM32 Read Out Protection (RDP)

󰃭 2024-01-02

In this blog post, I will cover all the steps I took to circumvent the Read Out protection mechanism. For those unfamiliar with this security measure deployed by ST to safeguard firmware and customer data, here is a brief explanation from their website:

  • https://www.st.com/resource/en/product_training/STM32F7_Security_Memories_Protections.pdf:

Read Out Protection

Readout Protection, or RDP is a global mechanism that prevents external read access to Flash memory, backup SRAM and registers. An external access can be gained by using a JTAG connector, a Serial Wire port or boot software embedded in SRAM. Three levels of RDP protection are defined from level 0, which offers no protection at all, to level 2 which has full and permanent protection. […] When first RDP level, Level 0, is set, the device has no protection. All read or write operations (if no write protection is set) on the Flash memory or the backup SRAM are possible in all boot configurations (Flash user boot, debug or boot from RAM). Option bytes are also changeable in this level.

RDP Levels :

  • Level 0 is is the factory default level.
  • In level 1, read protection is set for the Flash memory, the backup SRAM and the backup registers is set. In this level, protected memories are only accessible when booting from User Flash memory. Whenever a debugger access is detected or boot is not set to a Flash memory area, any access to the protected memories generates a system hard fault which blocks all code execution until the next power-on reset*
  • In level 2, all protections from Level 1, plus no SWD/JTAG connection possible, no going back. RDP/WRP option bytes can no longer be changed.

STM32 Bootloader

STM32 provides multiple options for accessing the flash memory. These protocols can be accessed either through the SWD interface or a built-in system bootloader, which is available in almost all STM32 devices at address 0x1FFF0000. The system bootloader allows users to interact with the STM32 device using various protocols to perform operations such as programming, erasing, or reading memory sectors. The protocols available include SWD, SPI, UART, and others.

Due to its simplicity, this discussion will focus on the UART protocol and how it interfaces with the bootloader.

Environment Setup

The environment consists of the following devices and tools:

  • STM32F401CC developer board
  • CW1173 ChipWhisperer-Lite
  • USB-UART TTL Converter
  • ST Link programmer

The device was programmed with a dummy firmware for testing and the Read Out Protection was set as shown in the following picture:

Before proceeding with the attack, it is important to determine how and when to glitch the device. The chosen technique for this attack is Voltage Fault Injection, which involves briefly dropping the device’s power supply just enough to cause it to skip instructions. Ideally, the glitch should occur precisely when the device is checking the Read Out Protection, resulting in either skipped instructions or a malformed data read.

To generate an effective voltage injection, it is very important to carefully review the target device and remove any potential sources of disruption to the glitch, such as voltage regulators, bypass capacitors, and similar components. Additionally, understanding the power domain of the device is fundamental to identifying the optimal point for injecting the glitch.

Power Supply schema

As observed, all VDD voltage is routed through an internal voltage regulator within the SoC. However, the pins VCAP_1 and VCAP_2 are directly connected to the core power domain, making them ideal targets for the voltage injection.

Hardware Preparation

  • A cable was soldered at the VCAP_1
  • The BOOT1 pin was manually forged to HIGH to have the device starting from the system bootloader at each reboot.
  • USART interface connected to the USB-UART TTL Converted
  • RESET pin connected to the Chipwhisperer TIO4

IMPORTANT: The external UART-TTL Converter is important because Chipwhisperer does not support UART parity configuration.

Execution

Once the hardware setup is properly configured, we are ready to determine the optimal glitch parameters.

However, before targeting the Read Out Protection function within the Bootloader, it is important to identify the minimum glitch settings required to induce a successful fault and cause the device to skip instructions. A simple routine like the one below can serve as a starting point:

[...]
		for (i = 0; i < 1000; i++) {
			for (j = 0; j < 1000; j++) {
				k++;
			}
		}
        print(i,k,j)
        [...]

This routine allows us to experiment with different glitch parameters and observe the effects, helping us fine-tune the settings for successful fault injection.

Identified, the parameters, repeat, width, offset, and clock frequency, we can proceed with the attack. For this particular attack, we will use the enable_only setting, which focuses solely on the repeat value and the frequency. The length of the glitch is calculated as 1/clkfreq * repeat. Another important parameter to consider is the ext_offset. This value defines the waiting period before initiating the glitch and is expressed in clock cycles as 1/clkfreq. During the attack, this parameter introduced several issues, which will be explained in more detail below.

To trigger the glitch immediately after the 0x11 read command is issued, the Chipwhisperer trigger pin was connected to the USART_TX line. However, we still need to determine the exact waiting time before initiating the glitch to ensure it occurs at the correct moment.

Determine the ext_offset

A logic analyzer was used to determine how long the command takes before being received. The logic analyzer software provides a measurement feature that is particularly useful in this case. This feature allows precise timing measurements of when the 0x11 command is transmitted and received

Different measures were taken resulting in the following estimation:

  • Command issue: 182.300 µs
  • Command response: 196.800 µs

We can conclude the minimum waiting time to issue a glitch can be calculate as following:

  • Clock cycle: 1/clkfreq== 10ns
  • Minimum Waiting time: 182300/10 = 18230 clock cycle.

ext_offset == number of clock cycles before issuing the glitch

Glitch Session

The initial run using a wide ext_offset range of 18230 to 19500 resulted in several responses. After analyzing the results, I determined that the glitch was causing both the Read-Out Protection and Write Protect features to have random values, resulting in a malformed environment.

After several sessions of fine-tuning and adjustments, the valid ext_offset parameter was found to be between 18300 and 18370. The script can now run:
